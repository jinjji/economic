## 개발 단계 개요

EconDigest는 YouTube 경제 채널 영상을 자동 수집·요약하여 Notion에 저장하고, 웹에서 열람하는 서비스입니다.
다음 5단계에 걸쳐 **계층적으로** 체계적으로 개발합니다.

### 🔄 개발 순서의 원칙

각 단계는 **이전 단계에 의존**합니다:
- **1단계** (골격) 없이는 2~5단계를 시작할 수 없음
- **2단계** (공통 모듈) 없이는 3단계부터 중복 코드가 많아짐
- **3단계** (핵심 기능) 완료 후에야 실제 동작하는 서비스가 됨
- **4단계** (추가 기능)는 있으면 좋지만, 3단계 완료 후 진행 가능
- **5단계** (배포)는 모든 기능이 안정화된 후 마지막에 진행

### 작업 목록
- [ ] Monorepo 디렉토리 구조 확립 (`pipeline/`, `web/`)
- [ ] Python 가상환경 설정 및 `requirements.txt` 작성
- [ ] Next.js 프로젝트 초기화 (App Router, TypeScript)
- [ ] 환경변수 파일 설정 (`.env.local.example` 작성)
- [ ] Git ignore 및 커밋 규칙 정의
- [ ] Notion API 접근 권한 설정 및 데이터베이스 생성

### 🤔 왜 이 순서인가?
**기반이 없으면 아무것도 할 수 없습니다.**

파이프라인과 웹이 공유해야 할 기본 구조(디렉토리, 설정, 환경변수, Notion 스키마)가 없으면:
- 양쪽이 다른 형식으로 작업하게 되어 나중에 통합이 어렵고
- 변경사항이 생기면 양쪽을 모두 수정해야 함

따라서 **모든 개발의 기반**을 먼저 확립하는 것이 가장 효율적입니다.

- 파이프라인과 웹이 같은 인터페이스를 사용해 통합이 쉬움
- 3단계에서는 비즈니스 로직(수집, 요약, UI)에만 집중 가능
- 나중에 Notion API가 변경되어도 한 곳만 수정하면 됨

## 3️⃣ 핵심 기능 개발

**목표**: **실제 동작하는 서비스** (MVP) 완성
- YouTube 수집 → AI 요약 → Notion 저장 (파이프라인)
- Notion 읽기 → 카드 목록 → 상세 페이지 (웹)

### 작업 목록

#### Pipeline (Python) - **필수: 수집·요약·저장**
- [ ] YouTube RSS 피드 파싱 (`src/services/youtube_service.py`)
- [ ] 자막 추출 (youtube-transcript-api)
- [ ] Claude API로 AI 요약 (`src/services/summarizer.py`)
- [ ] Notion DB 저장 (`src/services/notion_service.py` - write)
- [ ] 파이프라인 메인 흐름 통합 (`src/main.py`)
- [ ] 기본 에러 로깅

#### Web (Next.js) - **필수: 목록·상세 페이지**
- [ ] 홈 페이지: Notion 데이터 읽기 및 카드 목록
  - [ ] 기본 데이터 표시 (제목, 채널, 날짜, 요약)
  - [ ] 페이지네이션 또는 무한 스크롤
- [ ] 상세 페이지: 개별 카드 클릭 시 전체 정보 표시
  - [ ] 요약 전문
  - [ ] 키워드
  - [ ] 영상 링크

### 🤔 왜 이 순서인가?
**이 단계를 완료하면 실제 동작하는 서비스가 됩니다.**

파이프라인이 실제로 YouTube에서 데이터를 수집하고, 웹에서 그것을 볼 수 있어야만:
- 서비스의 핵심 가치(자동 수집+요약+조회)를 검증할 수 있음
- 데이터 흐름이 올바른지 테스트할 수 있음
- 기술 스택(Claude API, Notion API, Next.js)이 제대로 작동하는지 확인할 수 있음

필터, 검색 등 부가 기능은 이 기초 위에서 나중에 추가하면 됩니다.

### 예상 소요 시간
**5-7일**

### 완료 기준
- [ ] 파이프라인: YouTube 채널 5개 이상에서 영상 수집 및 요약 완료
- [ ] Notion DB에 100+ 영상 데이터 저장 (또는 테스트 데이터)
- [ ] 웹: 홈 페이지에서 모든 데이터 정상 표시
- [ ] 웹: 상세 페이지에서 각 영상의 전체 정보 확인 가능
- [ ] 공개여부 체크박스로 노출 제어 정상 작동

따라서 **핵심이 안정화된 후** 유연하게 추가할 수 있습니다.

## 📌 핵심 포인트

### 각 단계가 이전 단계를 필요로 하는 이유

```
1️⃣ 골격 (구조 & 환경)
  ↓ (없으면 → 개발 방향이 엇갈림)
2️⃣ 공통 모듈 (공유 코드)
  ↓ (없으면 → 중복 코드 증가, 통합 어려움)
3️⃣ 핵심 기능 (실제 서비스)
  ↓ (없으면 → 부가 기능이 붕괴할 기초가 없음)
4️⃣ 추가 기능 (사용성 개선)
  ↓ (없으면 → 배포 준비 안 됨)
5️⃣ 최적화 & 배포 (프로덕션)
```

### 중요한 의사결정
1. **Notion 스키마**: 1단계에서 정의하면 이후 수정 비용이 낮음
2. **API 비용**: Claude, YouTube, Notion API 호출량 모니터링
3. **3단계 이전 배포 금지**: 핵심 기능 없이 배포하면 나중에 아키텍처 변경이 어려움
4. **4단계는 선택**: 필터/검색 없이도 3단계까지 완료하면 서비스 가능
